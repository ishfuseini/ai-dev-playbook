# Worked Example — Fully Completed Template

> The following is a real, filled-in version of the template above.
> Use it as a reference when building your own.

---

## Prompt Identification

- **Name**: `generate-rest-endpoint`
- **Category**: Development
- **Use Case**: Use when adding a new REST API endpoint to an existing backend — generates the route, controller, service, and basic tests.

---

## Purpose and Goals

- **Primary Goal**: Generate a complete, production-ready REST endpoint including route, handler, service logic, and unit tests.
- **Expected Output**: Four files — `route.ts`, `controller.ts`, `service.ts`, and `service.test.ts` — following the project's existing patterns.
- **Not For**: GraphQL endpoints, WebSocket handlers, or batch jobs.

---

## S.C.A.F.F. Structure

### Situation
I'm working on a Node.js REST API built with Express and TypeScript. The codebase uses the Repository pattern for data access, a Service layer for business logic, and thin controllers for request/response handling. Authentication uses JWT middleware applied at the router level.

### Challenge
Create a new REST endpoint: `POST /api/v1/[resource]`. The endpoint should accept a JSON body, validate it with Zod, pass it to the service layer, persist it via the repository, and return a consistent API response envelope.

### Audience
The code will be maintained by a mid-level team familiar with TypeScript and Express but new to this specific codebase. Code should be readable without needing to reference other files.

### Format
- TypeScript with strict mode enabled
- Use Zod for input validation
- Follow the existing API response envelope: `{ success, data, error, meta }`
- Write unit tests with Vitest and mock the repository layer
- One export per file; no default exports

### Foundations
- Validate all inputs before they reach the service layer
- Never expose internal error details in API responses
- Use parameterized queries — no raw string interpolation in SQL
- Return 400 for validation errors, 409 for conflicts, 500 for unexpected errors
- Do NOT modify any existing files — only create new ones

---

## Notes & Learnings

- Including an existing endpoint as a reference example in "Code to Act On" significantly improves consistency with the codebase.
- Specifying the exact HTTP status codes per error type in Foundations prevents the AI from guessing.
- If the AI adds a default export, re-state the "no default exports" rule as a constraint.
