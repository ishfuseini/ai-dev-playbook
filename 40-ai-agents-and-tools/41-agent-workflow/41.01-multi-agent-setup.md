# Multi-Agent Setup

A guide for structuring AI agent roles in a development workflow. Define roles clearly before starting — ambiguity between agents causes duplicated work, conflicting changes, and poor output quality.

---

## Roles

### 🏛️ Architect
**Responsibility:** High-level planning, system design, and task decomposition. Produces the plan that all other agents execute against.

**Tasks:**
- Analyse requirements and define the technical approach
- Break work into phases and atomic tasks
- Define interfaces, data models, and system boundaries
- Review and update plans when new information changes scope

**Model Tier:** Premium — this is the most context-heavy role and mistakes here cascade downstream.

**Prompt / Instructions:**
```
You are a software architect. Your job is to plan, not implement.
Given a requirement, produce:
1. A phased breakdown of the work
2. Atomic tasks per phase, each completable independently
3. Clear interfaces and contracts between components
4. Explicit assumptions and open questions

Do NOT write implementation code. Output structured plans only.
```

---

### 👷 Developer
**Responsibility:** Implements tightly scoped tasks defined by the Architect. Works one task at a time.

**Tasks:**
- Implement a single, well-defined task
- Follow the plan exactly — flag deviations rather than making unilateral decisions
- Write tests alongside implementation
- Update the task log when complete

**Model Tier:**
- Routine / boilerplate tasks → Lightweight model (save tokens)
- Complex logic, security-sensitive, or ambiguous tasks → Premium model

**Prompt / Instructions:**
```
You are a senior developer implementing a specific task.
You will be given: a task description, relevant files, and constraints.

Rules:
- Implement ONLY what is described in the task
- Do NOT refactor unrelated code
- Do NOT make architectural decisions — flag them for the Architect
- Write or update tests for everything you implement
- When done, list: files changed, decisions made, and anything that needs review
```

---

### 🔍 Reviewer
**Responsibility:** Reviews completed work against the plan and coding standards. Acts as the quality gate before work is considered done.

**Tasks:**
- Verify implementation matches the plan
- Check for bugs, security issues, and code quality problems
- Validate test coverage
- Approve or return work with specific, actionable feedback

**Model Tier:** Premium — a cheap reviewer misses things that are expensive to fix later.

**Prompt / Instructions:**
See `44-claude/44.02-agents/code-reviewer.md` for the full reviewer agent definition.

---

### 📝 Documenter
**Responsibility:** Documents what was built after a task or phase is complete.

**Tasks:**
- Update `AI_CONTEXT.md` and relevant `AI_SUMMARY.md` files
- Write or update component documentation
- Log decisions made during implementation
- Update the Memory system task log

**Model Tier:** Lightweight — documentation is structured and low-risk; save tokens here.

**Prompt / Instructions:**
```
You are a technical writer documenting completed work.
You will be given: the task description, files changed, and decisions made.

Produce:
1. An updated AI_SUMMARY.md for any modified modules
2. Component documentation for any new components (use the component doc template)
3. A completed task log entry for the Memory system

Be concise and factual. Do not add opinions or suggestions — only document what exists.
```

## Model Capability Tiers

Rather than pinning to specific model names (which change frequently), use these tiers:

| Tier | When to Use | Examples |
|------|-------------|---------|
| **Premium** | Architect planning, security review, complex logic, code review | Claude Opus, GPT-4o, Gemini Ultra |
| **Standard** | Typical development tasks, moderate complexity | Claude Sonnet, GPT-4o-mini, Gemini Pro |
| **Lightweight** | Boilerplate, formatting, documentation, simple transforms | Claude Haiku, GPT-3.5, Gemini Flash |

**Rule of thumb:** Spend tokens where mistakes are expensive. Save tokens where output is easy to verify.

## Handoff Protocol

Clear handoffs prevent agents from working on stale or conflicting information.

```
Architect → Developer
  Output: Task file with goal, context, steps, constraints, and expected output
  Signal: "Task N.N is ready for implementation"

Developer → Reviewer
  Output: List of files changed, decisions made, and anything flagged for review
  Signal: "Task N.N implementation complete — ready for review"

Reviewer → Developer (if changes needed)
  Output: Categorised issues (Critical / Major / Minor) with specific file + line references
  Signal: "Task N.N returned — N critical issues to resolve before approval"

Reviewer → Documenter (if approved)
  Output: Approved task summary
  Signal: "Task N.N approved — ready to document"

Documenter → Architect
  Output: Updated docs, task log marked complete
  Signal: "Task N.N documented — Memory system updated"
```

## Orchestration Flow

```
                    ┌─────────────┐
                    │  Architect  │
                    │  (Plan)     │
                    └──────┬──────┘
                           │ Task file
                    ┌──────▼──────┐
                    │  Developer  │◄──────────────┐
                    │  (Build)    │               │
                    └──────┬──────┘               │
                           │ Files changed         │ Return with
                    ┌──────▼──────┐               │ issues
                    │  Reviewer   │───────────────►┘
                    │  (Check)    │
                    └──────┬──────┘
                           │ Approved
                    ┌──────▼──────┐
                    │ Documenter  │
                    │  (Record)   │
                    └──────┬──────┘
                           │ Docs updated
                    ┌──────▼──────┐
                    │  Architect  │
                    │ (Next task) │
                    └─────────────┘
```

**Human-in-the-loop checkpoints (recommended):**
- After Architect produces the plan — review before any code is written
- After Reviewer approves — before moving to the next phase
- Any time an agent flags a decision it cannot make autonomously

---

## When to Use a Single Agent vs. Multi-Agent

| Situation | Approach |
|-----------|---------|
| Small, well-understood task (<2 hours) | Single agent with clear prompt |
| Feature spanning multiple files or systems | Multi-agent: Architect + Developer + Reviewer |
| Ambiguous requirements | Start with Architect only until plan is solid |
| High-risk changes (auth, payments, data) | Full pipeline with Premium models at every stage |
| Documentation-only work | Documenter agent alone |
